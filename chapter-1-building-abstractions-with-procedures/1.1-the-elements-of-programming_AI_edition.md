## 全章导览

每种强大的语言都提供三种表达能力,用来**组合简单想法,构造复杂想法**:

1. **基本表达式 (primitive expressions)** — 语言最简单的实体
2. **组合手段 (means of combination)** — 从简单元素构建复合元素
3. **抽象手段 (means of abstraction)** — 给复合元素命名、把它们当作单元使用

程序处理两类"元素"—— **过程 (procedures)** 和 **数据 (data)**。本章只讲数值数据和处理它们的过程。

## 1.1.1 表达式 (Expressions)

Scheme 最基本的交互: 输入表达式,解释器回应它的值。

```scheme
486                  ; → 486
(+ 137 349)          ; → 486
(- 1000 334)         ; → 666
(* 5 99)             ; → 495
(/ 10 5)             ; → 2
(+ 2.7 10)           ; → 12.7
```

### 前缀表示法 (Prefix notation)

**运算符在前,操作数在后**,整体包在括号里: `(运算符 操作数1 操作数2 ...)`。

```scheme
(+ 21 35 12 7)       ; → 75  (可以多个操作数)
```

### 组合式可以嵌套

```scheme
(+ (* 3 5) (- 10 6)) ; → 19
```

没有长度限制。可读性靠换行和缩进(**pretty printing**):

```scheme
(+ (* 3
      (+ (* 2 4)
         (+ 3 5)))
   (+ (- 10 7)
      6))
```

## 1.1.2 命名与环境 (Naming and the Environment)

`define` 给值或过程起名字。

```scheme
(define size 2)
size                              ; → 2
(* 5 size)                        ; → 10

(define pi 3.14159)
(define radius 10)
(* pi (* radius radius))          ; → 314.159

(define circumference (* 2 pi radius))
circumference                     ; → 62.8318
```

`define` 是构造抽象最简单的手段: **用一个名字指代一个复合操作**,后面可以直接用名字。

解释器必须在某个**全局环境 (global environment)** 里维护"名字 → 值"的映射——这个环境是状态的一部分,第 3 章会讲透。

## 1.1.3 组合式的求值 (Evaluating Combinations)

求一个组合式的值,解释器按以下规则递归地工作:

1. 求出组合式各个子表达式的值
2. 把运算符 (最左元素) 应用到实际参数上

**关键点**: 这个规则是**递归的**——要评估组合式,先评估它的子组合式;子组合式又可能是组合式,再递归下去,直到遇到基本表达式 (数字、内建运算符、被 define 的名字)。

例子:

```scheme
(* (+ 2 (* 4 6))
   (+ 3 5 7))
;   ↓
;  ┌─ (+ 2 (* 4 6))  → (+ 2 24) → 26
;  │
;  └─ (+ 3 5 7)      → 15
;   ↓
;  (* 26 15) → 390
```

可视化这个过程,叶子节点是值,从叶子向树根"向上逼近"——这种叫 **tree accumulation**。

### 例外: 特殊形式 (special forms)

`define` 不走这个规则 (不能把名字当值传)。每个特殊形式 (`define`, `if`, `cond`, `lambda`, `quote`...) 有自己的求值规则。

## 1.1.4 复合过程 (Compound Procedures)

**过程定义**(procedure definition)让你为复合操作命名:

```scheme
(define (square x) (* x x))
```

- `square` = 过程名
- `x` = 形参 (formal parameter)
- `(* x x)` = 过程体

使用:

```scheme
(square 21)              ; → 441
(square (+ 2 5))         ; → 49
(square (square 3))      ; → 81
```

用过程定义新过程:

```scheme
(define (sum-of-squares x y)
  (+ (square x) (square y)))

(sum-of-squares 3 4)     ; → 25

(define (f a)
  (sum-of-squares (+ a 1) (* a 2)))

(f 5)                    ; → 136
```

数学上 `f(a) = (a+1)² + (2a)² = 5a² + 2a + 1`,代 a=5 得 136。

---

## 1.1.5 过程应用的代换模型 (Substitution Model)

### 代换模型是什么

把过程调用想象成: **用实参替换形参,然后求值过程体**。

以 `(f 5)` 为例:

```
(f 5)
→ (sum-of-squares (+ 5 1) (* 5 2))   ; 把 a 替换为 5
→ (sum-of-squares 6 10)
→ (+ (square 6) (square 10))
→ (+ (* 6 6) (* 10 10))
→ (+ 36 100)
→ 136
```

这是**直观的心智模型**。真实解释器内部并不真的"替换",但用这个模型理解求值不会出错——至少在引入赋值之前(第 3 章后会失效)。

### 应用序 vs 正则序 (Applicative vs Normal Order)

求值过程有两种顺序:

| | 应用序 (Applicative) | 正则序 (Normal) |
|---|---|---|
| 规则 | **先算参数,再代入** | **先全部代入,最后才算** |
| 对 `(f 5)` | `(+ 5 1)` 算 1 次 | `(+ 5 1)` 算 2 次 (因为 square 定义里 x 出现两次) |
| 使用者 | Lisp / Scheme / Python / Java / C | Haskell (惰性求值) |

### 正则序展开 `(f 5)`

```
(f 5)
→ (sum-of-squares (+ 5 1) (* 5 2))
→ (+ (square (+ 5 1)) (square (* 5 2)))
→ (+ (* (+ 5 1) (+ 5 1))                ← (+ 5 1) 被复制两份
     (* (* 5 2) (* 5 2)))                ← 同理
→ (+ (* 6 6) (* 10 10))
→ (+ 36 100)
→ 136
```

### 为什么 Lisp 选应用序

1. **效率**: 避免同一子表达式重复计算
2. **可预测性**: 一旦引入赋值/IO 这些副作用,正则序很难推理"这段代码到底跑了几次、什么时候跑"

**纯函数下两种顺序结果相同**; 有副作用或无限递归时会分化(见 Exercise 1.5)。

## 1.1.6 条件表达式和谓词 (Conditional Expressions and Predicates)

### 谓词 (Predicate) 是什么

**返回真假 (`#t` / `#f`) 的过程**。Scheme 惯例名字以 `?` 结尾。

```scheme
(> 5 3)        ; → #t
(even? 4)      ; → #t
(null? '())   ; → #t
```

### `cond` — 多分支

```scheme
(define (abs x)
  (cond ((> x 0) x)
        ((= x 0) 0)
        ((< x 0) (- x))))
```

语法: `(cond (<谓词1> <结果1>) (<谓词2> <结果2>) ... (else <兜底>))`,从上到下依次判断,第一个为真的分支的结果被返回。

### `if` — 二选一

```scheme
(define (abs x)
  (if (< x 0)
      (- x)
      x))
```

### `if` vs `cond`

| 维度 | `if` | `cond` |
|---|---|---|
| 分支数 | 固定 2 支 | 任意多支 |
| 语法 | 扁平 | 每支要括号 |
| 每支多语句 | 需 `begin` | 原生支持 |
| 用法 | 二选一 | 多选一 |

两者表达能力等价,选哪个看清晰度。两者都是**特殊形式**,都**只求值选中那一支**。

### 逻辑组合子: `and` / `or` / `not`

```scheme
(and e1 e2 ... en)   ; 短路:遇假返回假;全真返回最后一个值
(or  e1 e2 ... en)   ; 短路:遇真返回该值;全假返回 #f
(not e)              ; 真变假,假变真
```

`and` / `or` 是**特殊形式**,短路求值; `not` 是普通过程。

### `>=` 的两种等价实现

```scheme
(define (>= x y) (or (> x y) (= x y)))   ; x > y 或 x = y
(define (>= x y) (not (< x y)))          ; x 不小于 y
```

数学基础是实数的三歧律: `x > y`、`x = y`、`x < y` 三者恰好成立一个。所以:

$$x \geq y \iff (x > y) \lor (x = y) \iff \neg(x < y)$$

## 1.1.7 例子: 用牛顿法求平方根 (Square Roots by Newton's Method)

### 陈述性知识 vs 过程性知识 (What vs How)

数学定义:

$$\sqrt{x} = \text{那个 } y,\text{ 使得 } y \geq 0 \text{ 且 } y^2 = x$$

这是**陈述性** (declarative) 描述,告诉你"是什么",**不告诉你怎么算**。

伪 Lisp 翻译:
```scheme
(define (sqrt x)
  (the y (and (>= y 0) (= (square y) x))))
```
只是换个说法叫"那个 y",依然没回答"**怎么找**"——书里说 "this only begs the question" (循环论证)。

> **数学关心 what is,计算机关心 how to。过程必须是 effective (能有效执行)。**

### 牛顿法: 逐步逼近

**规则**: 若 $y$ 是 $\sqrt{x}$ 的猜测,则 $\frac{y + x/y}{2}$ 是更好的猜测。

**为什么有效**: 真实的 $\sqrt{x}$ 永远夹在 $y$ 和 $x/y$ 之间——猜小则商偏大,反之亦然。取平均必定更靠近。收敛速度是**二次收敛** (每步有效数字约翻倍)。

算 √2,从 y=1 开始:

| 猜测 y | 商 x/y | 平均 (y + x/y)/2 |
|---|---|---|
| 1 | 2 | 1.5 |
| 1.5 | 1.3333 | 1.4167 |
| 1.4167 | 1.4118 | 1.4142 |
| 1.4142 | ... | ... |

### Scheme 实现

```scheme
(define (sqrt x) (sqrt-iter 1.0 x))

(define (sqrt-iter guess x)
  (if (good-enough? guess x)
      guess
      (sqrt-iter (improve guess x) x)))

(define (improve guess x)
  (average guess (/ x guess)))

(define (average x y) (/ (+ x y) 2))

(define (good-enough? guess x)
  (< (abs (- (square guess) x)) 0.001))
```

测试:

```scheme
(sqrt 9)                         ; → 3.00009155413138
(sqrt (+ 100 37))                ; → 11.704699917758145
(sqrt (+ (sqrt 2) (sqrt 3)))     ; → 1.7739279023207892
(square (sqrt 1000))             ; → 1000.000369924366
```

### 四个核心思想

这节表面讲"求平方根",实际在演示 SICP 最重要的几个思想:

1. **陈述性 vs 过程性**: 过程必须回答 how,而非 what。
2. **递归 = 迭代**: 无需 `for`/`while`,函数自己调用自己就能表达循环。Scheme 控制结构极简。
3. **过程分解**: 5 个小过程各司其职 (`sqrt` / `sqrt-iter` / `improve` / `average` / `good-enough?`),名字即含义。
4. **逐步逼近 / 不动点**: 反复施加"改进函数"直到收敛。1.3 节会把它抽象成高阶函数 `fixed-point`。

### 细节坑

- **为什么用 `1.0` 而不是 `1`?** MIT Scheme 区分精确整数和小数。`10 / 6` = `5/3`(有理数),`10.0 / 6.0` = `1.6666...`。用 `1.0` 强制进入浮点运算。
- **`good-enough?` 的 bug (见 Exercise 1.7)**: 绝对容差 0.001 对极小数不够准、对极大数可能永不收敛。应改用**相对容差**。

### Exercise 1.6 — 为什么 `if` 必须是特殊形式

Eva 用 `cond` 写了个新 `if`:

```scheme
(define (new-if predicate then-clause else-clause)
  (cond (predicate then-clause)
        (else else-clause)))
```

简单测试完全正常:

```scheme
(new-if (= 2 3) 0 5)   ; → 5
(new-if (= 1 1) 0 5)   ; → 0
```

Alyssa 用它改写 `sqrt-iter`:

```scheme
(define (sqrt-iter guess x)
  (new-if (good-enough? guess x)
          guess
          (sqrt-iter (improve guess x) x)))
```

**结果: 死循环,栈溢出。**

#### 根本原因: 应用序 + 普通函数

- 原版 `if` 是**特殊形式**: 先算谓词,只算选中那支
- `new-if` 是**普通函数**: 应用序规则要求**所有参数**必须先算成值

在 `sqrt-iter` 里,`new-if` 的**参数 3** 是 `(sqrt-iter (improve guess x) x)`——递归调用自身。

```
调用 sqrt-iter
  ↓ 要算 new-if 的三个参数
  ├─ 参数 1: (good-enough? ...)  → #f
  ├─ 参数 2: guess                → 具体值
  └─ 参数 3: (sqrt-iter (improve ...) x)  ← 又是递归调用
         ↓ 进入新一层 sqrt-iter
           ↓ 又要算三个参数
             ↓ 参数 3 又是递归调用
               ↓ ...
```

**每一层都卡在"算参数 3",永远到不了"判断谓词"那一步**。谓词永远没机会拦住递归。栈爆炸。

#### 关键追问: 为什么里面的 `cond` 救不了?

`cond` 确实是特殊形式,但它**在 new-if 的函数体内部**。特殊形式的特殊待遇**只在函数体执行时起作用**,**参数求值阶段**依然按应用序规则——参数 3 永远算不完,`cond` 的特殊待遇永远轮不到用上。

**比喻**: cond 是有选择权的总经理,但他被锁在保险柜(普通函数)里。保安(应用序)在门口卡死访客(参数),总经理再聪明也见不到人。

#### 结论

> **递归终止依赖"某层能跳过递归分支"。**
> **"跳过"只有两条路: (1) 语法级别提供特殊形式 (Scheme 选这条); (2) 语言默认惰性求值 (Haskell 选这条)。**
> **Scheme 是应用序,所以 `if` 必须是特殊形式; `new-if` 两条路都占不上,在递归里必然死循环。**

### Python 验证

Python 跟 Scheme 同样是应用序,可以完美复现这个 bug:

```python
def new_if(predicate, then_clause, else_clause):
    if predicate:
        return then_clause
    else:
        return else_clause

def sqrt_iter_buggy(guess, x):
    return new_if(
        good_enough(guess, x),
        guess,
        sqrt_iter_buggy(improve(guess, x), x)   # ← 参数3,递归
    )

sqrt_iter_buggy(1.0, 9)   # → RecursionError: 栈溢出
```

跟书里预测的行为完全一致。

### Exercise 1.7 — `good-enough?` 的缺陷

**问题**:
- **极小数**: `(sqrt 0.0001)`,真实答案 0.01,但 0.001 容差**大于答案本身**,会误判"够好了"
- **极大数**: 浮点精度不足,`|guess² - x|` 可能永远达不到 0.001,循环**永不收敛**

**修法**: 用**相对容差**,看两次猜测之间的变化是否相对足够小:

```scheme
(define (good-enough? guess prev-guess)
  (< (abs (- guess prev-guess))
     (* guess 0.001)))
```

### Exercise 1.8 — 立方根

公式: 若 $y$ 是 $\sqrt[3]{x}$ 的猜测,更好的猜测是 $\frac{x/y^2 + 2y}{3}$。

```scheme
(define (cube-root x) (cbrt-iter 1.0 x))

(define (cbrt-iter guess x)
  (if (good-enough? guess x)
      guess
      (cbrt-iter (improve guess x) x)))

(define (improve guess x)
  (/ (+ (/ x (square guess))
        (* 2 guess))
     3))

(define (cube x) (* x x x))
(define (good-enough? guess x)
  (< (abs (- (cube guess) x)) 0.001))
```

**公式来源**: 牛顿法通用形式 $y_{new} = y - f(y)/f'(y)$。对 $f(y) = y^3 - x$,$f'(y) = 3y^2$:

$$y_{new} = y - \frac{y^3 - x}{3y^2} = \frac{2y^3 + x}{3y^2} = \frac{x/y^2 + 2y}{3}$$

公式不是凭空的,是牛顿法自动推出来的。

## 1.1.8 过程作为黑箱抽象 (Procedures as Black-Box Abstractions)

### 过程分解 (Procedural Decomposition)

sqrt 程序被分解成一组协作的小过程:

```
sqrt
  └─ sqrt-iter
       ├─ good-enough? ─── square, abs
       └─ improve ──────── average
```

**关键**: 分解不是随便切一刀。**每个过程要完成一件"有独立意义"的事**,名字能说清它做什么。反例是 `step1 / step2 / step3`——切开了但没有意义。

### 黑箱抽象 (Black-Box Abstraction)

**用的人只关心接口,不关心实现。**

写 `good-enough?` 的人不需要知道 `square` 怎么实现,只需要知道它接受一个数、返回它的平方。

**同一接口可以有多种实现**,彼此可互换:

```scheme
(define (square x) (* x x))                       ; 实现 A
(define (square x) (exp (double (log x))))        ; 实现 B (利用 x² = e^(2 ln x))
(define (double x) (+ x x))
```

对调用者而言两个 `square` 完全等价。**哪个更高效取决于硬件**——如果机器有高效的对数表,B 可能比 A 快。

### 好处

- **局部修改,全局受益**: 换实现不用动调用者
- **减轻认知负担**: 读代码时一次只想一层
- **协作分工**: 不同人基于接口独立开发
- **可替换性**: 库升级、切换算法都不影响上层

---

### 局部名字 (Local Names)

参数名是**实现者的选择**,不该影响调用者。这两个 `square` 应该无法区分:

```scheme
(define (square x) (* x x))
(define (square y) (* y y))
```

这意味着: **形参必须只在过程体内可见**——否则会出问题。

#### 灾难场景 (如果参数不是局部的)

```scheme
(define (good-enough? guess x) ...)   ; 用了 x
(define (square x) (* x x))           ; 也用了 x
```

如果两个 `x` 是同一个全局变量:

```
调用 (good-enough? 1.5 9)
  x = 9
  调用 (square 1.5)
    x = 1.5   ← 覆盖了外面的 9 !!!
    返回 2.25
  回到 good-enough?
  此时 x 是 1.5 (被污染了), 计算完全错
```

黑箱破功——`square` 的内部细节(用哪个名字)泄漏出来影响了 `good-enough?`。

#### 局部变量规则

Scheme 规定形参只在函数体内有效。每次函数调用进入一个独立的"小世界",跟外界同名变量互不干扰。

### 绑定变量 vs 自由变量

```scheme
(define (good-enough? guess x)
  (< (abs (- (square guess) x)) 0.001))
```

| 变量 | 类型 | 含义 |
|---|---|---|
| `guess`, `x` | **绑定 (bound)** | 当前过程的形参,只在过程体内有意义 |
| `square`, `abs`, `<`, `-` | **自由 (free)** | 函数体用到但不是本过程形参,指向外部定义 |

| | 改名后行为 | 依赖 |
|---|---|---|
| 绑定变量 | 可随便改名(一致地改),行为不变 | 函数本身定义 |
| 自由变量 | 改名意思会变 (比如把 `abs` 换成 `cos`) | 外部环境 |

### 变量捕获 (Variable Capture)

把 `guess` 改名成 `abs` 会踩坑:

```scheme
(define (good-enough? abs x)
  (< (abs (- (square abs) x)) 0.001))
         ↑
    这里本想调用"绝对值函数" abs,
    但 abs 现在是参数名,是一个数字!
    → 错误
```

原本是自由变量的 `abs` 被参数名**捕获**,从函数变成了一个数。改名要避免跟已用到的自由变量冲突。

### 作用域 (Scope)

**一个名字有效的代码范围**。形参的作用域 = 函数体。

### 内部定义和块结构 (Internal Definitions and Block Structure)

#### 平铺定义的两个麻烦

```scheme
(define (sqrt x) ...)
(define (sqrt-iter guess x) ...)
(define (good-enough? guess x) ...)
(define (improve guess x) ...)
```

1. **污染全局**: 用户只关心 `sqrt`,其他三个是实现细节却暴露在外
2. **名字冲突**: 如果另一个程序 `cube-root` 也想用 `good-enough?`,两份定义会冲突

#### 块结构 — 把辅助过程藏进主过程

```scheme
(define (sqrt x)
  (define (good-enough? guess x)
    (< (abs (- (square guess) x)) 0.001))
  (define (improve guess x)
    (average guess (/ x guess)))
  (define (sqrt-iter guess x)
    (if (good-enough? guess x)
        guess
        (sqrt-iter (improve guess x) x)))
  (sqrt-iter 1.0 x))
```

内部过程只在 `sqrt` 里可见,外部看不到——**完美封装**。不同程序可以各自有自己的 `good-enough?`,互不干扰。

**历史**: 块结构思想来自 **Algol 60**,现代几乎所有语言(Python/JS/C/Java...) 都继承了。

#### 词法作用域 (Lexical Scoping) — 再简化一步

注意 `x` 被反复当参数传,但它其实不变。让内部过程**直接用外层的 x**:

```scheme
(define (sqrt x)
  (define (good-enough? guess)          ; 不带 x
    (< (abs (- (square guess) x)) 0.001))
  (define (improve guess)                ; 不带 x
    (average guess (/ x guess)))
  (define (sqrt-iter guess)              ; 不带 x
    (if (good-enough? guess)
        guess
        (sqrt-iter (improve guess))))
  (sqrt-iter 1.0))
```

内部函数里的 `x` 是**自由变量**,词法作用域会按"文本所在的外层"找定义,于是自动找到 `sqrt` 的形参 `x`。

**规则**: **函数内部自由变量的含义,由它"写在哪"决定,而不是"谁调用它"决定**。这叫**词法作用域** (lexical scope / static scope)。对比"动态作用域"——后者按运行时调用链查找,现在几乎没人用。

#### 对比

| 版本 | 全局名字数 | 代码长度 |
|---|---|---|
| 平铺 | 5 个 (sqrt, sqrt-iter, good-enough?, improve, average) | 长,到处传 x |
| 块结构 | 1 个 (sqrt) | 短,x 只传一次 |

**块结构 + 词法作用域 = 把一个函数做成自包含的黑箱,内部细节对外完全不可见**。

### Python 对应

```python
def sqrt(x):
    def good_enough(guess):
        return abs(guess**2 - x) < 0.001       # x 是外层自由变量 (闭包)
    
    def improve(guess):
        return (guess + x / guess) / 2
    
    def sqrt_iter(guess):
        if good_enough(guess):
            return guess
        return sqrt_iter(improve(guess))
    
    return sqrt_iter(1.0)
```

这就是 Python 里的**闭包 (closure)**——嵌套函数 + 词法作用域的应用,跟 SICP 这里讲的完全是一回事。

## 本节核心思想总结

| 概念 | 一句话 |
|---|---|
| 代换模型 | 用实参替换形参,然后求值过程体——直观的心智模型 |
| 应用序 | 先算参数再代入; Lisp/Python/Java 用这个 |
| 正则序 | 先代入再用时算; Haskell 用这个 |
| 特殊形式 | `if`/`cond`/`and`/`or`/`define`/`lambda`/`quote` 有自己的求值规则,不走应用序 |
| 谓词 | 返回真假的过程,通常以 `?` 结尾 |
| 陈述性 vs 过程性 | "是什么" vs "怎么做"; 计算机需要"怎么做" |
| 牛顿法骨架 | 猜 → 改进 → 判断够不够好 → 没够好继续 |
| 黑箱抽象 | 用的人只看接口,不关心实现 |
| 局部名字 | 形参只在函数体内有效,互不污染 |
| 绑定 vs 自由变量 | 形参是绑定(可改名),引用外部的是自由(不可随便改) |
| 块结构 | 内部定义嵌套,隐藏辅助过程 |
| 词法作用域 | 自由变量在"文本外层"查找 = 闭包的基础 |

**1.1 是整本书的 "0 号基石"**——后面每一章都建立在这里讲的概念上。抽象 (abstraction)、递归 (recursion)、过程分解 (decomposition) 这三件事会被反复重写、加深、升华,贯穿整个 SICP。
