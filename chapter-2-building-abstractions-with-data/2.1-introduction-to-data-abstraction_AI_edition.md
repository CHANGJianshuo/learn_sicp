## 类比:过程抽象 vs 数据抽象

**它们是同一个思想方法论的两面**:

### 过程抽象(1.1.8 已讲)

```scheme
(define (square x) (* x x))
```

`square` 怎么实现的(乘法?exp+log?查表?)**对调用者无所谓**。**调用者只需要相信:输入 x,输出 x²**。

```
[调用者] ─────接口(square)───── [实现者]
不关心怎么算                      自由选择算法
```

### 数据抽象(2.1 要讲)

```scheme
(define x (make-rat 1 2))
(numer x)      ; → 1
(denom x)      ; → 2
```

`make-rat` 怎么"粘合"的(cons?list?lambda?字符串?)**对调用者无所谓**。**调用者只需要相信:能拿出分子和分母**。

```
[使用方] ─────接口(make-rat, numer, denom)───── [表示方]
不关心怎么存                                       自由选择存储方式
```

**对偶关系一目了然**:
- 过程抽象隐藏**算法细节**
- 数据抽象隐藏**表示细节**

## 数据抽象的运行机制

程序被分成两层,中间用接口隔开:

```
┌─────────────────────────────────┐
│  使用层 (use programs)          │   add-rat, sub-rat, mul-rat...
│  操作"抽象数据"                  │
├─────────────────────────────────┤
│  接口 (selectors + constructors)│   make-rat, numer, denom
├─────────────────────────────────┤
│  表示层 (representation)        │   pair, list, 或别的具体存储
└─────────────────────────────────┘
```

接口的两种角色:

| 名字 | 作用 | 例子 |
|---|---|---|
| **constructor (构造器)** | 把零件组装成复合数据 | `make-rat`、`make-point`、`make-segment` |
| **selector (选择器)** | 从复合数据里提取零件 | `numer`、`denom`、`x-point`、`y-point` |

构造器是"打包",选择器是"拆包"。**所有对数据的操作都通过这两类函数完成 — 绝不直接碰内部表示**。

## Pairs — Lisp 的"原子胶水"

> **整个 Lisp 语言的所有复合数据,从最简单的 list 到最复杂的 AST、tree、graph,都建立在三个内置过程上:`cons`、`car`、`cdr`。**

```scheme
(cons a b)   ; 把 a 和 b 粘成一个 pair
(car p)      ; 取出 pair 的第一个零件
(cdr p)      ; 取出 pair 的第二个零件
```

**核心契约**:

> **`(car (cons a b))` = a**
> **`(cdr (cons a b))` = b**

无论 a 和 b 是什么 — 数、字符串、布尔值、过程、甚至别的 pair — 这个契约永远成立。

### 闭包性 (Closure Property)

```scheme
(define x (cons 1 2))
(define y (cons 3 4))
(define z (cons x y))     ; cons 把两个 pair 又粘成新 pair
```

**`cons` 的输入可以是任何值;`cons` 的输出也是一个值;所以 `cons` 的输出可以再做 `cons` 的输入** — 没有任何深度限制。

**有了闭包性,只用 cons 一种胶水就够了** — 这是 Lisp 设计的极简之美。

### 名字的来历

`car` 和 `cdr` 来自 IBM 704(1957 年)的硬件指令:
- **CAR** = "Contents of the Address part of the Register"
- **CDR** = "Contents of the Decrement part of the Register"

McCarthy 1958 年写第一个 Lisp 时直接用了硬件名,**这两个名字保留了 60+ 年**。

## 2.1.1 有理数实例

### 三行接口

有理数 = 分子 + 分母,正好是一对数:

```scheme
(define (make-rat n d) (cons n d))   ; 构造器
(define (numer x) (car x))           ; 选择器:分子
(define (denom x) (cdr x))           ; 选择器:分母
```

**关键**:这三行就是**全部抽象屏障**。上层的 `add-rat / mul-rat` 只用这三个名字,**永远不直接 `cons / car / cdr`**。

### 加减乘除

按数学公式 $\frac{a}{b} + \frac{c}{d} = \frac{ad+bc}{bd}$:

```scheme
(define (add-rat x y)
  (make-rat (+ (* (numer x) (denom y))
               (* (numer y) (denom x)))
            (* (denom x) (denom y))))

(define (sub-rat x y)
  (make-rat (- (* (numer x) (denom y))
               (* (numer y) (denom x)))
            (* (denom x) (denom y))))

(define (mul-rat x y)
  (make-rat (* (numer x) (numer y))
            (* (denom x) (denom y))))

(define (div-rat x y)
  (make-rat (* (numer x) (denom y))
            (* (denom x) (numer y))))

(define (equal-rat? x y)
  (= (* (numer x) (denom y))
     (* (numer y) (denom x))))
```

**`add-rat` 等等的代码里没有一个 `cons / car / cdr`** — 全部通过领域接口操作。

### 打印

```scheme
(define (print-rat x)
  (newline)
  (display (numer x))
  (display "/")
  (display (denom x)))
```

### 测试发现的问题

```scheme
(define one-half  (make-rat 1 2))
(define one-third (make-rat 1 3))

(print-rat (add-rat one-half one-third))    ; → 5/6  ✓
(print-rat (mul-rat one-half one-third))    ; → 1/6  ✓
(print-rat (add-rat one-third one-third))   ; → 6/9  ✗ 应该是 2/3
```

最后一个**不化简**。1/3 + 1/3 数学上等于 2/3,但程序输出 6/9。

### 修复:在构造器里约分

```scheme
(define (make-rat n d)
  (let ((g (gcd n d)))
    (cons (/ n g) (/ d g))))
```

**关键**:**只改了一个函数(`make-rat`),`add-rat` / `mul-rat` 一行没动**。

> **抽象屏障的回报**:**修改影响范围被锁住**。所有跟"存储/构造"有关的事,只改构造器。

### Exercise 2.1 — 处理正负号

要求:正数时分子分母都正;负数时只让分子带负号。

```scheme
(define (make-rat n d)
  (let ((g (gcd (abs n) (abs d))))
    (if (< d 0)
        (cons (/ (- n) g) (/ (- d) g))    ; d 负 → n、d 同时取负
        (cons (/ n g) (/ d g)))))
```

逻辑:让分母永远是正的,分子的符号自然代表整个分数的符号。

**整个修复在 `make-rat` 里完成**,加减乘除一字未改 — 抽象屏障再次起作用。

> **数据进系统的入口 = 唯一的规范化点。** 数据库的字段验证、表单的格式化、API 的 schema 校验,本质上都是同一个思想。

## 2.1.2 抽象屏障的层次

### 屏障的层次图

```
┌──────────────────────────────────────────────┐
│  使用有理数的程序                              │
├──────────────────────────────────────────────┤  ← 屏障 1
│  有理数对外接口:                              │
│  add-rat / sub-rat / mul-rat /                │
│  div-rat / equal-rat?                         │
├──────────────────────────────────────────────┤  ← 屏障 2
│  有理数的构造器/选择器:                        │
│  make-rat / numer / denom                     │
├──────────────────────────────────────────────┤  ← 屏障 3
│  pair 接口:                                   │
│  cons / car / cdr                             │
├──────────────────────────────────────────────┤  ← 屏障 4
│  pair 的具体实现                               │
└──────────────────────────────────────────────┘
```

**水平线**就是抽象屏障。每一条线把上下两层"隔开" —— 上层只通过这条线提供的接口名字访问下层,**绝不直接看下层内部**。

### 为什么要分这么多层

#### 一、复杂度被切成小块

> **每写一段代码,只需要在脑里装下一层接口的概念,不必装下整个系统。**

#### 二、修改的"影响范围"被限定

每条屏障 = 一道"防火墙"。**屏障下面改,屏障上面就感觉不到**。

#### 三、设计阶段也受益

最容易被忽略的好处 — **抽象屏障让你能"暂时不决定"**。你不知道哪种实现更好?**没关系,先定接口,实现以后再说**。其他人可以基于接口先写上层代码,等下面的实现做出来再接进去。

**这就是大型团队并行开发的基础**。

### 同一个接口,两种实现

#### 实现 A:构造时约分

```scheme
(define (make-rat n d)
  (let ((g (gcd n d)))
    (cons (/ n g) (/ d g))))   ; 构造时就化简
(define (numer x) (car x))
(define (denom x) (cdr x))
```

#### 实现 B:访问时约分

```scheme
(define (make-rat n d) (cons n d))    ; 构造时不化简
(define (numer x)
  (let ((g (gcd (car x) (cdr x))))
    (/ (car x) g)))                    ; 取分子时才化简
(define (denom x)
  (let ((g (gcd (car x) (cdr x))))
    (/ (cdr x) g)))                    ; 取分母时才化简
```

| | 实现 A | 实现 B |
|---|---|---|
| 构造代价 | 高(每次造都算 gcd) | 低 |
| 访问代价 | 低 | 高(每次访问都算 gcd) |
| 适合场景 | **造一次,访问多次** | **造很多但很少访问** |

**两种都对**,选哪个取决于实际使用模式。**`add-rat`、`sub-rat` 一字不改**。

### Exercise 2.2 — 点 + 段 + 中点

#### 第一层:点

```scheme
(define (make-point x y) (cons x y))
(define (x-point p) (car p))
(define (y-point p) (cdr p))
```

#### 第二层:线段(用第一层做零件!)

```scheme
(define (make-segment start end) (cons start end))
(define (start-segment s) (car s))
(define (end-segment s) (cdr s))
```

**这里 cons 装的不是数字,而是两个点** — 这就是闭包性。

#### 中点

```scheme
(define (average a b) (/ (+ a b) 2))

(define (midpoint-segment s)
  (let ((p1 (start-segment s))
        (p2 (end-segment s)))
    (make-point (average (x-point p1) (x-point p2))
                (average (y-point p1) (y-point p2)))))
```

`midpoint-segment` 的代码里**没有一个 `cons / car / cdr`** — 全部通过领域接口操作。

### Exercise 2.3 — 矩形 + 抽象屏障真实考验

**核心挑战**:**实现两种不同表示**,要求**同一套 perimeter / area 跑两种实现都对**。

#### 表示 A:用两个对角点

```scheme
(define (make-rect bl tr) (cons bl tr))
(define (rect-bl r) (car r))
(define (rect-tr r) (cdr r))

(define (rect-width r)
  (abs (- (x-point (rect-tr r)) (x-point (rect-bl r)))))
(define (rect-height r)
  (abs (- (y-point (rect-tr r)) (y-point (rect-bl r)))))
```

#### 表示 B:用一个角点 + 宽 + 高

```scheme
(define (make-rect bl w h) (cons bl (cons w h)))
(define (rect-bl r)     (car r))
(define (rect-width r)  (car (cdr r)))
(define (rect-height r) (cdr (cdr r)))
```

#### 上层(perimeter / area)只用 width / height

```scheme
(define (rect-perimeter r)
  (* 2 (+ (rect-width r) (rect-height r))))

(define (rect-area r)
  (* (rect-width r) (rect-height r)))
```

**无论你换 A 还是换 B,这两个函数一字不改**。

#### 这道题的精髓

> **抽象屏障设计的关键:接口要用"领域概念"命名,不要泄漏"实现细节"。**

| 接口风格 | 后果 |
|---|---|
| 暴露表示细节(`rect-bl`、`rect-tr`) | 换表示就死 |
| 暴露领域概念(`rect-width`、`rect-height`) | 换表示无感 |

**好接口反映"使用方需要的概念",不是"实现方碰巧选的存储"**。

## 2.1.3 数据是什么?

### 第一步:数据 = 满足约束的接口

回顾有理数:三个函数 `make-rat / numer / denom`。**它们的契约**:

> **对任意整数 n 和非零整数 d,如果 x = (make-rat n d),那么 (numer x) / (denom x) 必须等于 n/d。**

这是这三个函数的**契约 (contract)**。只要满足这个契约,具体实现是什么都行。

> **数据 = 一组选择器和构造器 + 它们必须满足的条件。**

### 第二步:连 pair 自己也是一样

**pair 本身**呢?

> **对任意 x 和 y,如果 z = (cons x y),那么 (car z) 必须是 x,(cdr z) 必须是 y。**

**这是 cons / car / cdr 唯一的契约**。**任何**满足这个契约的三个函数,都可以叫"pair"。

### 震撼的演示:用 lambda 实现 pair

```scheme
(define (cons x y)
  (define (dispatch m)
    (cond ((= m 0) x)
          ((= m 1) y)
          (else (error "Argument not 0 or 1 -- CONS" m))))
  dispatch)

(define (car z) (z 0))
(define (cdr z) (z 1))
```

**完全没有数据结构**。`cons` 不分配任何"内存槽",它**返回一个函数**。

#### 一步步看

`(cons x y)` 返回的不是"装着 x 和 y 的盒子",**而是一个函数**(那个 dispatch),这个函数是个**闭包**,里面"封"着 x 和 y 的值。

`(car z)` = `(z 0)` = 把 z 这个函数应用到 0 上 → dispatch 返回 x。

`(cdr z)` = `(z 1)` = dispatch 返回 y。

**契约满足**。从外面看,这个 cons / car / cdr 跟内置的**完全无法区分**。

### 这意味着什么

> **数据可以是过程。**

`(cons 1 2)` 返回的"东西"内部其实是一段会执行的代码。但因为它满足 pair 的契约,**外面的代码完全看不出区别**。

### 消息传递 (Message Passing)

`(z 0)` 可以读作:**"给 z 发一个消息 0"** — z 收到消息后回应一个值。

这就是**对象 (object)** 的雏形:**一个东西,能根据收到的消息做出不同响应**。

第 3 章会大量使用这个模式实现**面向对象编程**:

```scheme
(define (make-account balance)
  (define (deposit amount) ...)
  (define (withdraw amount) ...)
  (define (dispatch msg)
    (cond ((eq? msg 'deposit) deposit)
          ((eq? msg 'withdraw) withdraw)))
  dispatch)
```

**Java / Python / Smalltalk 的"对象 + 方法"**,本质上就是这个 dispatch 函数。

### Exercise 2.4 — 另一种"过程版 pair"

```scheme
(define (cons x y)
  (lambda (m) (m x y)))

(define (car z)
  (z (lambda (p q) p)))

(define (cdr z)
  (z (lambda (p q) q)))
```

#### 用代换法验证 `(car (cons x y))` = x

```
(car (cons x y))
= (car (lambda (m) (m x y)))
→ ((lambda (m) (m x y)) (lambda (p q) p))
→ ((lambda (p q) p) x y)
→ x   ✓
```

#### 核心思想

`cons` 不"存"数据,它"封装一个动作":**"以后随便有人想用我的零件,只要给我一个'怎么用'的函数,我就把 x 和 y 喂给它。"**

`car` 给它"取第一个"的函数 `(lambda (p q) p)`;`cdr` 给它"取第二个"的函数 `(lambda (p q) q)`。

数学上这种构造叫 **Church pair**。

### Exercise 2.5 — 用整数实现 pair (哥德尔编码)

把 pair (a, b) 编码成整数 $2^a \cdot 3^b$。

```scheme
(define (cons a b)
  (* (expt 2 a) (expt 3 b)))

(define (car z) (count-divisions z 2))
(define (cdr z) (count-divisions z 3))

(define (count-divisions z base)
  (if (= 0 (remainder z base))
      (+ 1 (count-divisions (/ z base) base))
      0))
```

**算术基本定理保证**:每个正整数有唯一的素因数分解,所以 (a, b) → 整数是无歧义的、可逆的。

这是**哥德尔编码 (Gödel numbering)** 的雏形 — Gödel 1931 年用同样思路把"数学命题"编成整数,证明出**不完备性定理**。

### Exercise 2.6 — Church 数

> **不只 pair 不需要数据结构 — 连"数字"本身,语言都可以不提供。**

每个**数 n** 被编码成一个高阶函数:**"接收任意动作 f,把 f 重复做 n 次"**。

```scheme
(define zero (lambda (f) (lambda (x) x)))                    ; 不做 f
(define (add-1 n)
  (lambda (f) (lambda (x) (f ((n f) x)))))                    ; 多做一次

;; 直接定义:
(define one  (lambda (f) (lambda (x) (f x))))                ; 做一次
(define two  (lambda (f) (lambda (x) (f (f x)))))            ; 做两次
```

#### 加法

m + n = "施加 m+n 次 f" = "先施加 n 次,再施加 m 次":

```scheme
(define (church+ m n)
  (lambda (f)
    (lambda (x)
      ((m f) ((n f) x)))))
```

#### 验证 1 + 1 = 2

```
((church+ one one) f)
= (lambda (x) ((one f) ((one f) x)))
= (lambda (x) (f (f x)))
= two   ✓
```

### 三道题的真正教训

```
Ex 2.4: lambda 实现 pair  → 数据 = 函数
Ex 2.5: 整数实现 pair      → 数据 = 数(哥德尔编码)
Ex 2.6: lambda 实现数字本身 → 连"数字"都是函数
```

**每一题都在验证**:**"数据"是表象,任何能模拟出契约的东西都是数据**。

#### 历史 + 意义

**Alonzo Church 1936 年**用 lambda 演算证明:**纯 lambda 能表达所有可计算的东西**(Church-Turing 论题的一半)。

工程上没人用 Church 数(慢),思想上**这是 SICP 第 2 章最重要的洞察**:

> **过程和数据是同一种东西,只是人脑为了方便,起了两个名字。**

### 这一节的三个思想冲击

1. **数据的定义是"契约",不是"形状"** — 接口定义身份
2. **过程和数据的界限是约定** — 1.3 是过程是值,2.1.3 是数据是过程
3. **有了一等公民过程,就免费拥有"复合数据"** — 这是 Church-Turing 定理的微缩版

## 2.1.4 区间运算 — Exercises 2.7–2.16

围绕**区间 (interval)** 这个数据抽象,逐步加深 — 从最简单的接口,到揭示区间运算的根本困难。

### Exercise 2.7 — 完成接口

```scheme
(define (make-interval a b) (cons a b))
(define (lower-bound i) (car i))
(define (upper-bound i) (cdr i))
```

跟有理数完全一样的模式。

### Exercise 2.8 — 区间减法

`[a, b] - [c, d]` 的最小可能 = `a - d`,最大可能 = `b - c`:

```scheme
(define (sub-interval x y)
  (make-interval (- (lower-bound x) (upper-bound y))
                 (- (upper-bound x) (lower-bound y))))
```

### Exercise 2.9 — 加减的"宽度"性质

**宽度** = (上界 - 下界) / 2,衡量不确定性。

加法:`[a,b] + [c,d] = [a+c, b+d]`,新宽度 = `(b-a)/2 + (d-c)/2` = **两宽度之和**。

→ 加减的结果宽度只跟原宽度有关 ✓

**乘法不行**:`[1,2] × [1,2]` 宽度 0.5,结果 `[1,4]` 宽度 1.5;`[10,11] × [10,11]` 宽度 0.5,结果 `[100,121]` 宽度 10.5 — **同宽度,结果宽度差几十倍**。

### Exercise 2.10 — 跨零除法报错

```scheme
(define (div-interval x y)
  (if (and (<= (lower-bound y) 0) (>= (upper-bound y) 0))
      (error "Division by interval spanning zero" y)
      (mul-interval x
                    (make-interval (/ 1.0 (upper-bound y))
                                   (/ 1.0 (lower-bound y))))))
```

根据 4 个端点的正负号(各 3 种),共 9 种组合,大多数只需 2 次乘法。**符号已知就能预知最大最小在哪两个角**。

### Exercise 2.12 — 中心 + 百分比表示

工程师习惯写 `100Ω ± 5%` 而不是 `[95, 105]`:

```scheme
(define (make-center-percent c p)
  (make-center-width c (* c (/ p 100.0))))

(define (percent i)
  (* 100.0 (/ (width i) (center i))))
```

**抽象屏障的胜利**:加新构造器/选择器,**底层 `make-interval` 一字不动**。

### Exercise 2.13 — 乘法的近似容差

两个**正区间**乘法,中心相乘、容差**近似相加**:

$$(c_1 \pm c_1 p_1) \times (c_2 \pm c_2 p_2) \approx c_1 c_2 \pm c_1 c_2 (p_1 + p_2)$$

(展开后忽略 $p_1 p_2$ 二阶小项。) `5% × 3%` 的结果约 `8%`。

### Exercise 2.14–2.16 — 揭示根本问题

#### Ex 2.14:同一公式两种写法,不同结果

并联电阻公式:

```
par1:  R1·R2 / (R1+R2)
par2:  1 / (1/R1 + 1/R2)
```

**用区间运算跑,结果不一样**。`R1 = R2 = 100Ω ± 1%`:
- `par1`: 50 ± ~3%
- `par2`: 50 ± ~1%

#### Ex 2.15:Eva 说 par2 "更好"

她的观察:**par2 里 R1、R2 各只出现一次;par1 里都出现两次**。

每出现一次,区间运算就**重新累加一次不确定性** — 但实际上 R1 还是那个 R1。**重复出现 = 重复累加 = 结果区间被人为放大**。

#### Ex 2.16:为什么 + 能否修复

**根本原因**:区间运算**忘记变量身份**。`A - A` 数学上 = 0,但区间算 `[a,b] - [a,b] = [a-b, b-a]`(非零)。

**能不能修复?** 理论上几乎不可能。这是区间算术的**本质局限**。

### 整组练习的真正教训

1. **2.7-2.12**:数据抽象 + 多种表示 — 抽象屏障的力量再次验证
2. **2.13**:近似公式让你看到"工程估算"如何从严格数学里来
3. **2.14-2.16**:**抽象不是万能的**

> **数据抽象帮你管理"实现细节",但管不了"算法本身的根本性质"。**
> **`A - A` 在区间运算里就是 ≠ 0,这是区间算术的本质,不是 bug,改不掉。**

## 本节核心思想总结

| 概念 | 一句话 |
|---|---|
| 数据抽象 | 把"数据怎么用"和"数据怎么存"分开 |
| 构造器 / 选择器 | 接口的两类函数:打包 + 拆包 |
| 抽象屏障 | 程序里画水平线,上下两层独立演化 |
| pair / cons / car / cdr | Lisp 唯一的胶水,所有复合数据的原子 |
| 闭包性 | pair 进 cons,出来的还是能再 cons 的东西 |
| 数据的本质是契约 | 任何满足契约的东西都是合法数据 |
| 用 lambda 实现 pair | 过程和数据的界限是约定 |
| Church 数 | 数字也是函数 — "重复某动作 N 次"的能力 |
| 消息传递 | 数据 = 能响应消息的函数,OOP 的雏形 |
| 抽象的边界 | 区间算术的"变量身份丢失"问题 — 抽象救不了算法本质 |

## 实践的反射

学完 2.1,以后看任何数据时给自己问:

1. **这堆数据是被一起处理的吗?** → 把它们粘成一个整体,用接口隔离
2. **接口里是不是泄漏了实现细节?** → 用领域词汇命名,而不是表示词汇
3. **修改时是不是只改一处就行?** → 影响范围越窄,屏障设计越成功
4. **底层换实现,上层会不会受影响?** → 设计抽象屏障的终极测试

## 历史 + 思想意义

**SICP 在 1980 年代用一个有理数例子,把数据抽象的核心思想讲透了**。

这一节的方法论后来变成整个软件工程的基础:
- Java 的 interface
- C++ 的虚函数 / 抽象基类
- Haskell 的 type class
- TypeScript 的 type
- Go 的 interface
- 微服务的 API 契约
- 数据库的 schema

**全都是 data abstraction 在不同语言/不同尺度下的体现**。

而 2.1.3(用 lambda 实现 pair、Church 数)则**指向一个更深的真理** —— **过程和数据的边界是约定不是本质**。这条路一直通到第 4 章"实现 Scheme 解释器",会让你彻底改变对"程序是什么"的理解。

## 一句话总结全节

> **2.1 的核心动作 = 在程序里画一道屏障,让"数据怎么用"和"数据怎么存"独立演化。**
